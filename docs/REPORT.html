1문제 정의 & 전략 (Why)

문제 (Problem)

AI 에이전트 메모리는 append-only다. 낡고 모순된 지식이 누적 → 컨텍스트 오염 + 토큰 비용 폭증. 핵심 결핍은 "기억"이 아니라 "망각(forgetting)"이다.

전략 (Strategy)

토큰 하네스(harness) 자체를 프로덕트로 승격. 5개 선행 개념을 조합: LLM Wiki(karpathy, 유지보수되는 마크다운 지식) + autoresearch(자동 증류) + graphify(링크 그래프) + codegraph(토큰 절약형 인덱스) + AI-DLC(단위작업 U1–U10 워크플로).

수상 요건의 계층화 (의사결정 프레임): L0 Pass/Fail Qwen 사용·오픈소스·3분 영상·마감 준수 — L1 득점 60% 기술축: 측정된 토큰 효율 + 재현성 — L2 차별화 망각 엔진 + 정직한 벤치 리포트 + MCP 가점.

2시스템 아키텍처 (What)

토큰 절감 기제 ①

index-first top-3 검색: 전체 위키 스캔 대신 index.md 기반 최소 컨텍스트만 주입.

토큰 절감 기제 ②

small-first 라우팅: qwen-flash 우선, 신뢰도 미달 시에만 qwen-plus로 escalate-on-fail.

토큰 절감 기제 ③

정적 prompt prefix(캐시 적중) + 엄격 JSON 출력 계약(freeform 대비 출력 토큰 절반).

3개발 프로세스: AI-DLC 단위작업 U1–U10 (How)

4검증 결과: A/B 벤치마크 (Evidence)

방법: 질문 12개 × 3실험, 성공기준 = 유효 JSON + 실존 위키 페이지를 가리키는 인용 ≥1 (엄격). KPI = tokens_per_success (원시 토큰이 아닌 "성공한 답변당 비용").

실험베이스라인tok/success Δ성공률
L-E1HEAVY + 전체 위키 컨텍스트−47.1%11/12 vs 6/12
L-E2HEAVY 전용 (동일 검색)−17.2%11/12 vs 7/12
L-E3LIGHT + freeform (계약 없음)−51.3%11/12 vs 12/12*

* L-E3 베이스라인 성공은 "비어있지 않은 텍스트"만 요구(인용 검증 불가) — 토큰 2.2배 쓰면서 근거는 0.

정직한 리포팅 (핵심 학습): L-E2에서 후보의 원시 토큰은 +30% 더 든다 — LIGHT 실패 시 escalate로 호출이 2회가 되기 때문. 그러나 heavy-only는 엄격 계약을 더 자주 실패(7/12)하므로 성공당 비용에선 여전히 −17% 우위. 부정적 수치를 숨기지 않는 것 자체가 "eval 기반 엔지니어링"의 증거이며 심사 전략상 우위다. 원시 토큰 최소화가 아니라 cost_per_success 최소화가 올바른 목적함수.

6핵심 학습 포인트 (Lessons)

범주교훈
제품 전략수상 요건을 L0/L1/L2로 계층화하면 "무엇을 하고 무엇을 버릴지"가 명확해진다. Docker는 요구사항이 아니었지만 90% 완료 상태라 "버그 수정 후 동결"이 정답 — 매몰비용이 아니라 재현성 가치로 판단.
측정 설계성공기준을 느슨하게 두면(텍스트 존재 여부) 벤치가 무의미해진다. "인용이 실존 페이지를 가리키는가"로 강화하자 결과의 설득력이 오히려 상승. 소논문은 불필요, 재현 가능한 벤치 리포트가 적정 수준.
토큰 공학절감의 3레버 = 컨텍스트 최소화(index-first) · 모델 라우팅(small-first) · 출력 계약(JSON). 단, 라우팅은 소규모 코퍼스에서 오버헤드 가능 — 코퍼스가 클수록 이득 증가.
환경 함정Windows: PowerShell BOM이 LICENSE 감지를 깨뜨림(UTF8Encoding($false)로 해결) · Docker에서 package=false 프로젝트는 ENV PYTHONPATH 필수 · MCP rpds DLL 오류는 lazy import로 회피.
프로세스AI-DLC 단위작업(U1–U10)이 진행 추적·동결 시점 판단·커밋 단위를 일관되게 만들었다. "신규 기능 금지 동결 원칙"이 제출 품질을 지킨다.
